# Plan: Fix TaskMateTelegramBot Issues

## Problem Analysis

Based on code exploration, the following issues were identified:

### Issue 1: Task assignee information not displayed
- **Location**: Telegram bot message formatting functions
- **Root Cause**: `task_list_item_text()` and `task_detail()` in `TaskMateTelegramBot/src/bot/messages.py` do not include assignee information, even though the API returns task assignments with user data.
- **Impact**: Users cannot see who is assigned to tasks in the bot interface.

### Issue 2: Intermittent visibility of open shifts
- **Location**: Shift fetching logic in Telegram bot
- **Potential Root Causes**:
  - API status filtering may not account for all "open" shift statuses (e.g., "late" shifts might be considered open but filtered out)
  - Dealership scoping might incorrectly filter shifts for owners (though code shows owners bypass scoping)
  - Possible race conditions in shift status updates or API response caching
  - Timezone issues in shift start/end time comparisons
- **Impact**: Managers and owners sometimes cannot see open shifts they should have access to.

## Proposed Solutions

### Solution 1: Display Task Assignees
**Files to modify**:
- `TaskMateTelegramBot/src/bot/messages.py` - Update `task_list_item_text()` and `task_detail()` functions

**Changes**:
- Extract assignee names from `task['assignments']` array
- Format assignee names appropriately for display
- Add assignee information to both task list items and detailed views
- Handle cases where assignments may be empty or contain multiple assignees

**Implementation Steps**:
1. Create helper function to format assignee names from assignments array
2. Update `task_list_item_text()` to include assignee summary (e.g., "Assigned to: John Doe, Jane Smith")
3. Update `task_detail()` to show full assignee list
4. Test with tasks having single and multiple assignees

### Solution 2: Fix Shift Visibility Issues
**Investigation needed**:
- Check what shift statuses are considered "open" in the system
- Verify dealership scoping logic for owners vs managers
- Examine if there are any caching issues in the API responses
- Check for timezone-related filtering problems

**Files to examine**:
- `TaskMateServer/app/Http/Controllers/Api/V1/ShiftController.php` - Index method filtering
- `TaskMateServer/app/Services/DealershipAccessService.php` - Scoping logic
- `TaskMateServer/app/Enums/ShiftStatus.php` - Available shift statuses

**Potential Fixes**:
1. **Status Filtering**: Update shift query to include all relevant "open" statuses (open, late, etc.)
2. **API Debugging**: Add logging to track what shifts are returned for owners
3. **Caching Check**: Ensure no caching layers are interfering with real-time shift visibility
4. **Timezone Handling**: Verify shift time comparisons use consistent timezones

**Implementation Steps**:
1. Review ShiftStatus enum and confirm which statuses should be visible as "open"
2. Update ShiftController index method to include appropriate statuses
3. Add debug logging to track shift filtering for owners
4. Test with various shift statuses and user roles
5. Verify dealership scoping doesn't incorrectly filter owner access

## Testing Strategy
- Test task assignee display with individual and group tasks
- Test shift visibility for owners, managers, and employees
- Create test shifts with different statuses to verify filtering
- Test across different dealerships to ensure proper scoping

## Risk Assessment
- Low risk for task assignee changes (UI improvement only)
- Medium risk for shift visibility changes (could affect performance if query becomes too broad)
- Need to ensure backward compatibility with existing bot users

## Implementation Order
1. Implement task assignee display (quick win, low risk)
2. Investigate and fix shift visibility issues (requires more analysis)</content>
